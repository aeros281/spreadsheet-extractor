# Releasing

## Overview

Releases are fully automated via [`.github/workflows/releases.yml`](../.github/workflows/releases.yml). Pushing a semver tag triggers cross-platform builds, changelog generation, and a draft GitHub Release. The only manual step is publishing the draft.

## Prerequisites

- Write access to the repository (to push tags).
- All changes intended for the release are merged to `master`.
- Commits follow [Conventional Commits](https://www.conventionalcommits.org) — `feat:`, `fix:`, `refactor:`, `doc:`, `perf:`, `style:`, `test:`. Non-conventional commits are filtered out of the changelog by `cliff.toml`.

## Steps

### 1. Choose a version

Follow [Semantic Versioning](https://semver.org):

| Change type | Bump |
|---|---|
| Breaking CLI or config change | major (`X.0.0`) |
| New subcommand or flag (backwards-compatible) | minor (`x.Y.0`) |
| Bug fix, refactor, docs only | patch (`x.y.Z`) |

### 2. Update `Cargo.toml` and `Cargo.lock`

```toml
[package]
version = "X.Y.Z"
```

Then run `cargo build` (or any cargo command) so `Cargo.lock` is updated to reflect the new version. Stage both files and commit:

```bash
git add Cargo.toml Cargo.lock
git commit -m "chore(release): prepare for vX.Y.Z"
```

This commit is automatically skipped by git-cliff (matched by `chore(release): prepare for` in `cliff.toml`), so it won't appear in the changelog.

### 3. Push the tag

```bash
git tag vX.Y.Z
git push origin vX.Y.Z
```

The tag must match `v[0-9]+.[0-9]+.[0-9]+*` to trigger the workflow.

### 4. Wait for CI

The workflow runs two jobs:

1. **`build_artifacts`** — cross-compiles the binary for all 9 targets in parallel and uploads them as GitHub Actions artifacts. Targets: FreeBSD x86\_64, Linux x86\_64/aarch64/arm/riscv64 (musl), Windows x86\_64/aarch64, macOS x86\_64/aarch64.

2. **`create_release`** — after all builds succeed, checks out the full git history, runs `git cliff --latest` to generate release notes from commits since the previous tag, then creates a **draft** GitHub Release with the binaries attached and the changelog as the body.

### 5. Publish the draft

Go to the repository's **Releases** page, review the draft, and click **Publish release**.

## Changelog generation

Release notes are generated by [git-cliff](https://git-cliff.org) using [`cliff.toml`](../cliff.toml). Only commits since the previous tag are included. Commits are grouped as:

| Prefix | Section |
|---|---|
| `feat` | Features |
| `fix` | Bug Fixes |
| `refactor` | Refactor |
| `doc` | Documentation |
| `perf` | Performance |
| `style` | Styling |
| `test` | Testing |
| `chore`, `ci` | Skipped |

To preview what the next release notes will look like locally (requires `git-cliff` installed):

```bash
git cliff --latest --strip header
```

## Manual trigger

The workflow can also be triggered manually from the GitHub Actions UI via `workflow_dispatch` without pushing a tag. In that case the `create_release` job is skipped (it requires a tag ref), so only the build artifacts are produced.
